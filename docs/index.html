pgflows Durable workflow engine for Python + Postgres

Write long-running, fault-tolerant workflows as plain async Python functions backed entirely by your existing Postgres database. No extra infrastructure, no separate orchestration service. State, retries, and scheduling all live in the DB.

Get started Step types PyPI

What you get

The full lifecycle of a workflow system without adding new infrastructure to operate.

▶

Checkpoint replay

Workflows survive crashes. Completed steps are cached and never re-executed on worker restart.

🔒

Typed end-to-end

Step inputs and outputs are Pydantic models. No dict[str,Any] at any boundary.

🔄

Two execution modes

Pull worker (Python orchestrates via pgmq) or push mode (pg_durable orchestrates, calls your steps via HTTP or pgmq+NOTIFY).

🔌

Plugin hooks

before_workflow, after_workflow, on_workflow_error, before_step, after_step, on_step_error. Zero changes to workflow code.

⏱

Configurable retries

Per-step RetryConfig with exponential or linear backoff, jitter, and per-workflow defaults.

📊

Execution history

pg_durable records the full per-run trail. Typed Pydantic access via instance_nodes, instance_executions, metrics.

🕐

Cron scheduler

Recurring workflows via PgCronBackend, backed by pg_durable's df.wait_for_schedule. No pg_cron needed.

📤

SQL export

Export any Python workflow to a pg_durable SQL DSL string. Deploy without code changes or compose at runtime from config.

🔧

Swappable backends

Every infrastructure concern lives behind an ABC. Swap queue, orchestrator, or scheduler without touching workflow code.

Quick Start

Docker Compose + one import + two decorators is the full getting-started story.

1. Install
$uv add pgflows
#FastAPI push-mode endpoints:
$uv sync --extra fastapi
2. Start Postgres (pgmq + pg_durable pre-installed)
$docker compose up -d
#or pull directly:
$docker pull niradler/pgflows-postgres:17-1.5.1-0.2.2
3. First workflow
import asyncio
from pydantic import BaseModel
from pgflows import PgflowsConfig, RetryConfig, StepContext, WorkflowApp, WorkflowContext

app = WorkflowApp(config=PgflowsConfig(
    dsn="postgresql://pgflows:pgflows@127.0.0.1:5433/pgflows_test",
    otel_enabled=False, db_ssl=False,
))

class OrderInput(BaseModel):
    order_id: str
    amount: float

class OrderResult(BaseModel):
    charged: bool
    confirmation: str

@app.step(retry=RetryConfig(max_retries=3, initial_delay_seconds=1.0))
async def charge_payment(ctx: StepContext, input: OrderInput) -> OrderResult:
    return OrderResult(charged=True, confirmation=f"CHG-{input.order_id}")

@app.workflow()
async def process_order(ctx: WorkflowContext, input: OrderInput) -> OrderResult:
    return await ctx.step(charge_payment, input)

async def main() -> None:
    await app.initialize()                                        # migrations + pools
    iid = await app.start(process_order, OrderInput(order_id="ORD-1", amount=99.0))
    await app.process_once()                                      # pump one poll batch
    status = await app.get_status(iid)
    print(status.state, status.output)                            # COMPLETED OrderResult(...)
    await app.close()

asyncio.run(main())

Step Types

pgflows supports five distinct step kinds. Workflow and step functions are defined the same way across both execution modes; what changes is how pg_durable or the Python worker invokes them.

1
Python Step (@app.step())
Pull mode -- checkpointed execution

A regular async Python function registered with @app.step(). Called via await ctx.step(fn, input) inside a workflow. Its result is persisted to Postgres before the next step runs, making the workflow idempotent: if the worker crashes and restarts, the step is not re-executed -- the cached result is returned immediately.

  • +Signature: (StepContext, InputModel) -> OutputModel
  • +Per-step RetryConfig with exponential/linear backoff
  • +Optional timeout_seconds per step
  • +Plugin hooks fire before/after each execution (not on replay)
from pgflows import RetryConfig, StepContext

@app.step(
    name="charge_payment",          # optional, defaults to fn name
    retry=RetryConfig(
        max_retries=3,
        backoff="exponential",
        initial_delay_seconds=1.0,
    ),
    timeout_seconds=30.0,
)
async def charge_payment(ctx: StepContext, inp: OrderInput) -> OrderResult:
    # ctx.instance_id, ctx.step_name, ctx.attempt available
    result = await payment_api.charge(inp.order_id, inp.amount)
    return OrderResult(charged=True, confirmation=result.id)
2
Workflow Function (@app.workflow())
Pull mode -- the durable unit of work

The workflow function is the top-level orchestrator. It calls steps via await ctx.step(fn, input), which checkpoints each result. The workflow itself is the durable unit: it is enqueued to pgmq, a worker picks it up, and executes it in full (replaying completed steps from cache). Return a BaseModel to populate get_status().output.

  • +Signature: (WorkflowContext, InputModel) -> OutputModel | None
  • +Any async Python logic is valid inside a workflow (if-else, loops, etc.)
  • +step_defaults=RetryConfig(...) applies to all steps
  • +Started via app.start(fn_object, input) -- pass the function, not a string
from pgflows import WorkflowContext, WorkflowState

@app.workflow(
    name="process_order",
    step_defaults=RetryConfig(max_retries=2),
)
async def process_order(ctx: WorkflowContext, inp: OrderInput) -> OrderResult:
    validation = await ctx.step(validate_order, inp)
    if not validation.ok:
        return OrderResult(charged=False, confirmation="REJECTED")
    charge = await ctx.step(charge_payment, inp)
    await ctx.step(send_confirmation, charge)
    return charge

# Start and observe
iid = await app.start(process_order, OrderInput(...))
await app.process_once()
status = await app.get_status(iid)
if status.state is WorkflowState.COMPLETED:
    result = OrderResult.model_validate(status.output)
3
HTTP Step (df.http() via FastAPI)
Push mode -- pg_durable calls your endpoint

In push mode, pg_durable is the orchestrator. It calls your Python step via HTTP using df.http(). The pgflows FastAPI router registers one POST endpoint per @app.step() function at /pgflows/steps/{step_name}. pg_durable passes the step input as the JSON body and the instance ID via the X-DF-Instance-ID header.

  • +Same @app.step() function, no extra code needed
  • +pg_durable durably retries the HTTP call on failure
  • +Graph operators (&, |, |=>) apply to HTTP steps
  • !Requires an inbound HTTP server; use pgmq binding if no server available
from pgflows.fastapi_integration import create_pgflows_router

# Mount once -- registers /pgflows/steps/{name} for every @app.step()
router = create_pgflows_router(app, prefix="/pgflows")
fastapi_app.include_router(router)

# Compose a graph using http() DSL builder
from pgflows import http
node = (
    http(f"{BASE_URL}/pgflows/steps/validate_order",
         body='{"order_id":"ORD-1","amount":99.0}',
         headers={"Content-Type": "application/json",
                  "X-DF-Instance-ID": "{sys_instance_id}"})
    .capture("v")
    >> http(f"{BASE_URL}/pgflows/steps/charge_payment",
            body='{"order_id":"ORD-1","amount":99.0}',
            headers={"Content-Type": "application/json"})
)
iid = await app.pg_durable.start(node, label="order")
4
pgmq+NOTIFY Step (app.worker_step())
Push mode -- no inbound HTTP server needed

An alternative push-mode binding: pg_durable sends the step to a pgmq queue and fires a pg_notify. A Python StepWorker (started with app.run_step_worker()) reads the message, runs the step, and writes the result to pgflows.worker_step_results. pg_durable polls that table (race-free) rather than using df.wait_for_signal.

  • +No inbound HTTP server required -- pure outbound connections
  • +Use app.worker_step() not bare worker_step() to bind app config
  • +Thread data between steps via capture= / input_expr="$cap::jsonb"
  • +Parallel joins of worker_step branches resolve reliably
import asyncio

# Build a pipeline: double_it -> add_ten (passing result via capture)
node = (
    app.worker_step("double_it", capture="d")
    >> app.worker_step("add_ten", input_expr="$d::jsonb", capture="r")
)

# First step reads from the "input" durable variable
await app.pg_durable.setvar("input", '{"n": 4}')

# Start the StepWorker to drain the queue
asyncio.create_task(app.run_step_worker())

iid = await app.pg_durable.start(node, label="pipeline")

# Wait and read result
status = await app.pg_durable.status(iid)   # 'completed'
result = await app.pg_durable.result(iid)   # {"rows":[{"result":28}]}
5
SQL Node (sql_node())
Push mode -- pure SQL, no Python

A sql_node() wraps a raw SQL statement that pg_durable executes directly. No Python code runs -- the query is evaluated inside Postgres. The first column of the first row is the node's result, capturable via .capture("name"). Write single quotes naturally (WHERE status = 'pending'); the builder doubles them for the SQL-literal layer automatically.

  • +No Python process needed -- runs entirely in Postgres
  • +Composable with all DSL operators (>>, &, |=>, etc.)
  • +Can read/write any table in the same database
  • !Joins of trivial SQL nodes may hang on the bundled pg_durable build; prefer worker_step branches for joins
from pgflows import sql_node

# Simple audit write after a step completes
node = (
    app.worker_step("process_data", capture="r")
    >> sql_node("INSERT INTO audit(result) SELECT result FROM pgflows.worker_step_results WHERE id = (SELECT max(id) FROM pgflows.worker_step_results)")
)

# Capture a count and branch on it
check = sql_node("SELECT count(*) FROM orders WHERE status = 'pending'").capture("n")
node = check >> sql_node("SELECT 'has_work'").if_then(
    sql_node("SELECT 1"),   # count > 0: process
    sql_node("SELECT 0"),   # count = 0: skip
)

iid = await app.pg_durable.start(node, label="check")

pgmq: How It Works

pgmq (by Tembo) implements Postgres-native message queues as regular tables. pgflows uses it for two independent concerns: workflow dispatch in pull mode and step dispatch in push mode.

Message lifecycle

Each queue is backed by a table pgmq.q_{queue_name}. Messages are JSON payloads with a unique message ID and a visibility timeout (VT) lock.

  • 1 send -- message lands in the queue, immediately visible
  • 2 read (with VT) -- message becomes invisible to other readers for VT seconds. One reader gets it; concurrent readers skip it.
  • 3a delete (ack) -- message is permanently removed. pgflows does this on success.
  • 3b archive (dead-letter) -- message moves to pgmq.a_{queue}. pgflows does this on failure.
  • 3c VT expires -- if neither delete nor archive happens within VT seconds, the message becomes visible again for re-delivery.
stateDiagram-v2 [*] --> Visible: pgmq.send Visible --> Locked: pgmq.read (VT starts) Locked --> Deleted: pgmq.delete (success) Locked --> Archived: pgmq.archive (dead-letter) Locked --> Visible: VT expires (re-delivery) Deleted --> [*] Archived --> [*]

The two queues pgflows uses

Workflow queue (pull mode)
Queue: pgflows_workflows
VT config: workflow_visibility_timeout_seconds
send: app.start(fn, input) enqueues the workflow
read: WorkflowWorker polls, gets exclusive lock via VT
claim: atomic pending → running prevents duplicate processing when multiple workers race
delete: workflow completes successfully
archive: workflow fails after all retries exhausted
Step queue (push mode)
Queue: pgflows_steps
NOTIFY channel: pgflows_steps
VT config: step_visibility_timeout_seconds
send: pg_durable calls pgmq.send when reaching a worker_step node
notify: pg_notify wakes the StepWorker immediately
execute: StepWorker reads, runs the Python step
result: writes to pgflows.worker_step_results (race-free)
poll: pg_durable polls the results table until the row appears

Why poll a results table instead of df.wait_for_signal?

A NOTIFY-woken StepWorker can signal back to pg_durable before pg_durable has registered the waiter for that instance node. That signal is dropped and the instance hangs forever. The poll table is race-free: the result row persists until pg_durable reads it, regardless of timing.

Operations (SQL -- useful for debugging and ops)

-- List all queues and their sizes
SELECT * FROM pgmq.list_queues();

-- Queue depth and oldest message age
SELECT * FROM pgmq.metrics('pgflows_workflows');
SELECT * FROM pgmq.metrics('pgflows_steps');

-- Inspect dead-lettered workflow messages
SELECT msg_id, message, archived_at
FROM pgmq.a_pgflows_workflows
ORDER BY archived_at DESC
LIMIT 20;

-- Peek at queued messages without locking them
SELECT msg_id, message, enqueued_at
FROM pgmq.q_pgflows_workflows
ORDER BY enqueued_at
LIMIT 10;

-- Drain a stuck step queue (drops all messages -- use with care)
SELECT pgmq.purge_queue('pgflows_steps');

-- Inspect step results table
SELECT * FROM pgflows.worker_step_results ORDER BY created_at DESC LIMIT 20;

pg_durable: Graph Execution, History & Parallelism

pg_durable (by Microsoft, prefix df.) is a Postgres extension that implements durable, graph-based workflow orchestration entirely inside the database. It stores the full execution graph, runs it node by node (or in parallel), and records the complete history.

How pg_durable executes a workflow

When you call df.start(dsl_expr, label), pg_durable:

  1. 1 Parses the DSL expression into an execution graph and stores it in df.instances with status pending.
  2. 2 The executor picks up the instance and starts executing nodes. Status becomes running.
  3. 3 Each node's result is recorded in df.instance_nodes. Structural nodes (THEN, JOIN, IF) each get their own row.
  4. 4 For & (join), both branches are dispatched concurrently. The executor waits for ALL to reach completed before advancing.
  5. 5 If the executor or process restarts, it replays from the last completed node. Already-completed nodes return their cached result instantly.
flowchart TD A[df.start DSL] --> B[Parse graph, store in df.instances] B --> C[Executor picks up instance] C --> D{Node type?} D -->|sequence ~>| E[Run left, then right] D -->|join &| F[Dispatch both branches concurrently] F --> G[Wait for ALL to complete] G --> H[Advance past join] D -->|conditional ?>/!>| I[Evaluate, pick branch] D -->|loop @>| J[Re-enter body until break] E --> K[Record in df.instance_nodes] H --> K I --> K J --> K K --> L{More nodes?} L -->|yes| D L -->|no| M[Instance completed]

Parallelism with & (join) and | (race)

Join (&) -- wait for ALL

Both branches run concurrently as independent sub-graphs. The join node advances only when every branch has status completed. Captures made inside branches are available after the join.

# Fan out: run both steps in parallel, then continue
node = (
    (
        app.worker_step("fetch_users", capture="u")
        & app.worker_step("fetch_orders", capture="o")
    )
    >> sql_node("INSERT INTO audit VALUES (now())")
)
# $u and $o are both available after the join
Race (|) -- first wins

Both branches start. The first to complete provides the result; the other continues running but its result is ignored. Race does not cancel the loser and is reliable only as a terminal node (do not sequence after a race).

# Race two data sources -- first to respond wins
# NOTE: only use | as a terminal node
node = (
    sql_node("SELECT data FROM fast_cache WHERE key = 'x'")
    | sql_node("SELECT data FROM slow_store WHERE key = 'x'")
)
# Good: terminal. Bad: node >> next_step (hangs)
⚠
Parallelism caveats (bundled pg_durable build)
Joins of trivial bare-SQL branches (sql_node("SELECT 1") & sql_node("SELECT 2")) can hang on the bundled build (branches complete, JOIN stays running). Joins of worker_step branches resolve reliably. Also: a loop cannot share an instance with a parallel node (ContinueAsNew deadlock) -- put the loop-drain and the parallel checks in separate df.start instances.

Execution history

pg_durable records the full per-run trail in the database. pgflows exposes it as typed Pydantic models via PgDurableClient.

client = app.pg_durable   # requires app.pg_durable_available

# High-level instance metadata
info = await client.instance_info(iid)
# InstanceInfo fields:
#   label, function, status, output, created_at, updated_at

# Per-node execution trail
nodes = await client.instance_nodes(iid)
# list[InstanceNode] -- one row per graph node, including
# structural THEN/JOIN/IF rows the extension inserts.
# Fields: node_type, query, result_name, status, result, updated_at
for n in nodes:
    print(n.node_type, n.status, n.result)

# Timing and event counts
execs = await client.instance_executions(iid)
# list[ExecutionRecord]
# Fields: status (Title-case "Completed"), duration_ms, event_count
for e in execs:
    print(e.status, e.duration_ms, e.event_count)

# Cluster-wide counters
m = await client.metrics()
# Metrics fields: total, running, completed, failed, cancelled
SQL -- direct inspection
-- All running instances
SELECT * FROM df.list_instances('running');

-- Full node trail for one instance
SELECT * FROM df.instance_nodes('inst-id-here');
-- Returns structural rows: THEN, JOIN, IF, LOOP in addition
-- to your actual node rows. More rows than nodes you wrote.

-- Instance execution log (timing + events)
SELECT * FROM df.instance_executions('inst-id-here');

-- Cancel a stale running instance
SELECT df.cancel('inst-id-here', 'reason');

-- Cluster metrics
SELECT * FROM df.metrics();

Signals, loops, and conditional branching

Signals -- external event coordination

Suspend an instance until an external event (human approval, webhook, another service) sends a signal. The captured result is the full envelope; read your payload under ->'data'.

from pgflows import wait_for_signal

# Build the graph: wait then branch on approval
decision = wait_for_signal("approval").capture("d")
node = decision >> sql_node(
    # $d::jsonb->'data'->>'approved' is your payload field
    "SELECT ($d::jsonb->'data'->>'approved')::boolean"
).capture("ok")

iid = await app.pg_durable.start(node, label="approval-flow")

# From another process: send the signal
await app.pg_durable.signal(
    iid, "approval", {"approved": True}
)
Loops -- poll until done

Infinite loop with loop(body), while-loop with loop(body, cond). Exit with break_() inside an if_rows check.

from pgflows import loop, break_, if_rows, sql_node

# Poll a table until a row appears
body = (
    sql_node("SELECT id FROM jobs WHERE status='done' LIMIT 1")
    .capture("job")
    >> if_rows("job",
        break_(),                            # found: exit loop
        sql_node("SELECT df.sleep(5)"),      # not found: wait
    )
)
node = loop(body)

# Do NOT put a parallel node in the same instance as a loop --
# run them as separate df.start() calls to avoid deadlock.
Conditional branching

Branch on the result of any node using .if_then(true_branch, else_branch) or the standalone if_node() / if_rows().

from pgflows import if_node, if_rows, sql_node

# Branch on whether a SQL query returns rows
check = sql_node(
    "SELECT id FROM orders WHERE status = 'pending' LIMIT 1"
).capture("pending")

node = check >> if_rows(
    "pending",
    app.worker_step("process_order"),   # has rows
    sql_node("SELECT 'nothing to do'"), # no rows
)

# Branch on a boolean expression
flag = sql_node("SELECT count(*) > 0 FROM flags WHERE name='ff'")
node = if_node(
    flag,
    app.worker_step("new_path"),
    app.worker_step("old_path"),
)

from pgflows import LoggingPlugin, PgflowsPlugin, StepEvent, WorkflowEvent

app.register_plugin(LoggingPlugin())   # built-in: logs all events

class MetricsPlugin(PgflowsPlugin):
    async def after_step(self, event: StepEvent, result: object) -> None:
        metrics.record("step.completed", tags={
            "step": event.step_name,
            "attempt": event.attempt,
        })

    async def on_step_error(self, event: StepEvent, error: Exception) -> None:
        metrics.record("step.failed", tags={"step": event.step_name})

    async def on_workflow_error(self, event: WorkflowEvent, error: Exception) -> None:
        alerting.notify(f"Workflow {event.workflow_name} failed: {error}")

app.register_plugin(MetricsPlugin())   # register BEFORE initialize()
await app.initialize()

from pgflows import RetryConfig

@app.step(retry=RetryConfig(

SQL Export

Export any Python workflow to a pg_durable SQL DSL string. Deploy without code changes, compose at runtime from config.

from pgflows import SqlExporter

exporter = SqlExporter(registry=app.registry, base_url="https://api.example.com/pgflows")

# Export one workflow -- produces runnable pg_durable SQL
sql = exporter.export_workflow("process_order")

# Dry-run: inspect steps without SQL
result = exporter.dry_run("process_order")   # DryRunResult
for s in result.steps:
    print(s.step_name, s.http_url)

# Compose at runtime from a step list (no Python fn required)
sql = exporter.compose(
    workflow_name="on_call_response",
    steps=["page_engineer", "create_ticket"],
)
# compose() raises KeyError immediately on unregistered step names

# Export all registered workflows
sql_all = exporter.export_all()

# Select binding mode
http_sql   = app.exporter(base_url="https://...", mode="http").export_workflow("process_order")
worker_sql = app.exporter(mode="worker").export_workflow("process_order")
Binding modes
mode="http" (default): pg_durable calls your FastAPI endpoint via df.http().
mode="worker": pg_durable enqueues via pgmq and StepWorker runs the step.
AST-based step discovery
Step names are found by AST introspection. await ctx.step(fn_name, inp) is detected. await ctx.step(lookup(name), inp) is not.

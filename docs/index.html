pgflows Durable workflow engine for Python + Postgres

Write long-running, fault-tolerant workflows as plain async Python functions backed entirely by your existing Postgres database. No extra infrastructure, no separate orchestration service, no new runtime to operate. All state, retries, and scheduling live in the DB.

Get started Execution modes PyPI

What you get

pgflows gives you the full lifecycle of a workflow system without adding new infrastructure to operate.

▶

Checkpoint replay

Workflows survive crashes. Completed steps are cached in Postgres and never re-executed on worker restart.

🔒

Typed end-to-end

Step inputs and outputs are Pydantic models. No dict[str, Any] at any boundary. Serialization goes through model_dump_json / model_validate_json.

🔄

Two execution modes

A Python pull worker polling pgmq, or push mode where pg_durable orchestrates the graph and calls your steps via HTTP or pgmq+NOTIFY.

🔌

Plugin hooks

before_workflow, after_workflow, on_workflow_error, before_step, after_step, on_step_error. Add metrics, logging, or alerting without touching workflow code.

⏱

Configurable retries

Per-step RetryConfig with exponential or linear backoff and optional jitter. Set defaults at the workflow level and override per step.

📊

Execution history

pg_durable records the full per-run trail. Access it as typed Pydantic models via instance_nodes, instance_executions, metrics.

🕐

Cron scheduler

Recurring workflows via PgCronBackend, backed by df.wait_for_schedule. No pg_cron extension required.

📤

SQL export

Export any Python workflow to a pg_durable SQL DSL string. Deploy without code changes, compose at runtime from config, or inspect step sequences before running.

🔧

Swappable backends

Every infrastructure concern is behind an ABC. Swap the queue, orchestrator, or scheduler without touching a single workflow or step definition.

Architecture

WorkflowApp is the single entry point. It wires backends, the registry, and the worker. The rest of the system only talks to ABCs defined in backends/base.py.

2 @app.workflow() + @app.step() decorators register functions
3 await app.initialize() opens pools + runs migrations
Queue layer pgmq

Quick Start

Docker Compose + one import + two decorators is the full getting-started story.

1. Install
$uv add pgflows
#or pip install pgflows
#FastAPI push-mode endpoints:
$uv sync --extra fastapi
2. Start Postgres (pgmq + pg_durable pre-installed)
$docker compose up -d
#or pull the pre-built image:
$docker pull niradler/pgflows-postgres:17-1.5.1-0.2.2
3. Write your first workflow
import asyncio
from pydantic import BaseModel
from pgflows import PgflowsConfig, RetryConfig, StepContext, WorkflowApp, WorkflowContext

config = PgflowsConfig(
    dsn="postgresql://pgflows:pgflows@127.0.0.1:5433/pgflows_test",
    otel_enabled=False,
    db_ssl=False,
)
app = WorkflowApp(config=config)


class OrderInput(BaseModel):
    order_id: str
    amount: float


class OrderResult(BaseModel):
    charged: bool
    confirmation: str


@app.step(retry=RetryConfig(max_retries=3, initial_delay_seconds=1.0))
async def charge_payment(ctx: StepContext, input: OrderInput) -> OrderResult:
    # call your payment API here
    return OrderResult(charged=True, confirmation=f"CHG-{input.order_id}")


@app.workflow()
async def process_order(ctx: WorkflowContext, input: OrderInput) -> OrderResult:
    return await ctx.step(charge_payment, input)


async def main() -> None:
    await app.initialize()                                            # runs migrations, opens pools
    iid = await app.start(process_order, OrderInput(order_id="ORD-1", amount=99.0))
    await app.process_once()                                          # pump one poll batch
    status = await app.get_status(iid)
    print(status.state, status.output)                                # COMPLETED OrderResult(...)
    await app.close()

asyncio.run(main())

Execution Modes

pgflows supports two distinct orchestration models. Choose one per workflow; step definitions are shared between both.

Concern Pull (Worker) Push (pg_durable)
Orchestrator Python WorkflowWorker polling pgmq pg_durable in Postgres (df extension)
Durability Python-side checkpointing via pgmq + PgStateBackend pg_durable native replay, graph stored in DB
Step invocation await ctx.step(fn, input) in Python df.http() to FastAPI, or pgmq+NOTIFY to StepWorker
Parallelism Sequential within a workflow by default Native & (join), | (race) graph operators
Best for Python-heavy logic, complex branching, quick setup SQL-native graphs, cross-DB, signals, scheduled flows

Pull mode: Python orchestrates

@app.workflow(name="process_order")
async def process_order(ctx: WorkflowContext, inp: OrderInput) -> None:
    result = await ctx.step(validate_order, inp)
    await ctx.step(charge_card, ChargeInput(order_id=result.order_id))

@app.step()
async def validate_order(ctx: StepContext, inp: OrderInput) -> ValidationResult:
    ...

# Start + observe
iid = await app.start(process_order, OrderInput(...))   # pass the FUNCTION
await app.process_once()                                # pump one batch
status = await app.get_status(iid)
if status.state is WorkflowState.COMPLETED:
    out = MyOutput.model_validate(status.output)

# Blocking worker loop (use asyncio.create_task for background)
await app.run_worker()
ℹ
WorkflowState is UPPERCASE
WorkflowState.COMPLETED, PENDING, RUNNING, FAILED, CANCELLED, SUSPENDED. Different from PgDurableClient.status() which returns lowercase strings.
ℹ
Pass the function, not a string
app.start(process_order, inp) takes the function object. Passing a string raises AttributeError.
✓
Dead-letter included
Failed workflows are archived to pgmq.a_{queue} instead of being requeued forever.

Push mode: pg_durable orchestrates

from pgflows.fastapi_integration import create_pgflows_router

# Mount FastAPI step endpoints (df.http() calls these)
router = create_pgflows_router(app, prefix="/pgflows")
fastapi_app.include_router(router)

# pgmq+NOTIFY binding (no inbound HTTP needed)
node = (
    app.worker_step("double_it", capture="d")
    >> app.worker_step("add_ten", input_expr="$d::jsonb", capture="r")
)
asyncio.create_task(app.run_step_worker())   # drain queue in background
iid = await app.pg_durable.start(node, label="pipeline")

# Use http() for direct HTTP step calls
from pgflows import http
node = (
    http(f"{base_url}/pgflows/steps/validate_order",
         body='{"order_id":"ORD-1"}',
         headers={"Content-Type": "application/json"})
    .capture("v")
    >> http(f"{base_url}/pgflows/steps/charge_card",
            input_expr="$v::jsonb")
)
⚠
Use app.worker_step(), not worker_step()
The bare worker_step() builder hardcodes queue name pgflows_steps. If you rename the queue in config, it silently enqueues to a queue no worker drains and the instance hangs.
⚠
Thread data with captures, not setvar
With more than one durable var set, pg_durable serializes the snapshot with non-deterministic key order. A parallel join replay then fails. Use one config var and pass everything else via capture= / input_expr="$cap::jsonb".
ℹ
Needs both extensions in one DB
The default compose image ships only pgmq. Build the push-mode test image for both: docker build -t pgflows-e2e-dfpgmq:latest tests/e2e/docker

How pgmq is used

pgflows uses pgmq for two independent concerns: workflow dispatch (pull mode) and step dispatch in push mode. The SDK shares a single asyncpg connection pool across both.

sequenceDiagram participant A as User Code participant Q as pgmq queue participant W as WorkflowWorker participant C as WorkflowContext participant DB as PgStateBackend A->>Q: pgmq.send (workflow message) W->>Q: pgmq.read (poll batch) Q-->>W: message + payload W->>C: execute workflow fn C->>DB: checkpoint step result DB-->>C: cached result (on replay) W->>Q: pgmq.delete (ack) W->>Q: pgmq.archive (on failure)
Workflow queue (pull mode)
  • Queue name: pgflows_workflows (configurable)
  • Visibility timeout: workflow_visibility_timeout_seconds
  • Failed messages archived to pgmq.a_{queue}
  • Atomic pending→running claim prevents races
Step queue (push mode)
  • Queue name: pgflows_steps (configurable)
  • NOTIFY channel: pgflows_steps
  • StepWorker drains via app.run_step_worker()
  • Results written to pgflows.worker_step_results (poll-based, race-free)

DSL Builders

Python builders that mirror pg_durable SQL operators. Import from pgflows. Compose with Python operators (>>, &, |), then pass the result to app.pg_durable.start().

Python pg_durable SQL Description
a >> ba ~> bSequence: run b after a completes
a & ba & bParallel join: run both, wait for ALL to complete
a | ba | bRace: first to complete wins. Reliable only as a terminal node (does not cancel loser)
node.capture("x")node |=> 'x'Capture node's first-column result as $x for downstream use
node.if_then(t, e)node ?> t !> eConditional: branch on node's result
sql_node("SELECT 1")'SELECT 1'Wrap raw SQL (auto-escapes single quotes for the SQL-literal layer)
sleep(30)df.sleep(30)Sleep for N seconds
wait_for_signal("ok")df.wait_for_signal('ok')Suspend until signal arrives. Captured result is the full {signal_name, timed_out, data} envelope; read payload under ->'data'
wait_for_schedule("0 * * * *")df.wait_for_schedule('0 * * * *')Suspend until the next cron tick
loop(body)@> (body)Infinite loop. Combine with break_() to exit
loop(body, cond)df.loop(body, cond)While-loop: repeat while condition holds
break_()df.break()Exit the enclosing loop
if_node(cond, t, e)cond ?> t !> eConditional branch as a standalone node
if_rows("x", t, e)df.if_rows('x', t, e)Branch on whether capture $x has rows
join3(a, b, c)df.join3(a, b, c)Three-way parallel join
http(url, method, body, headers)df.http(...)HTTP step call. headers is a dict (not JSON string). Include Content-Type: application/json.
app.worker_step("name", ...)pgmq.send ~> pg_notify ~> poll ~> readRun a Python step via pgmq+NOTIFY, poll result table. Use app.worker_step (not bare worker_step) to bind app config.
Composing a graph
from pgflows import sql_node, sleep, wait_for_signal, loop, break_, if_rows

# Sequence: validate then charge
node = sql_node("SELECT 1") >> sql_node("SELECT 2")

# Parallel fan-out then continue
node = (
    sql_node("SELECT count(*) FROM users").capture("u")
    & sql_node("SELECT count(*) FROM orders").capture("o")
) >> sql_node("INSERT INTO audit(msg) VALUES ('done')")

# Wait for external signal then branch
decision = wait_for_signal("approval").capture("d")
node = decision >> if_rows(
    "d",
    sql_node("SELECT 'approved'"),
    sql_node("SELECT 'rejected'"),
)

# Pass the composed node to pg_durable
iid = await app.pg_durable.start(node, label="my-graph")
worker_step pipeline (pgmq+NOTIFY)
# Prefer app.worker_step() over bare worker_step()
node = (
    app.worker_step("double_it", capture="d")
    >> app.worker_step("add_ten", input_expr="$d::jsonb", capture="r")
)

# First step needs its input seeded before start:
await app.pg_durable.setvar("input", '{"n": 4}')
iid = await app.pg_durable.start(node, label="pipeline")

# Run the step consumer
asyncio.create_task(app.run_step_worker())

# Read the result
status = await app.pg_durable.status(iid)  # 'completed'
result = await app.pg_durable.result(iid)  # parsed JSON
⚠
Write single quotes naturally in sql_node() (WHERE status = 'pending'). The builder doubles them for the SQL-literal layer. Do not pre-double them yourself.

Plugin System

Extend behavior via lifecycle hooks without touching workflow code. Plugin errors are swallowed and never abort steps.

Available hooks
HookWhen
before_workflowBefore the workflow function is called
after_workflowAfter the workflow completes successfully
on_workflow_errorWhen the workflow raises an exception
before_stepBefore each step executes (not on replay)
after_stepAfter each step completes successfully
on_step_errorWhen a step raises an exception
StepEvent fields
event.instance_id
event.workflow_name
event.step_name
event.step_index
event.attempt
Custom plugin example
from pgflows import LoggingPlugin, PgflowsPlugin, StepEvent, WorkflowEvent

# Built-in: logs all lifecycle events
app.register_plugin(LoggingPlugin())

# Custom: implement any subset of hooks
class MetricsPlugin(PgflowsPlugin):
    async def after_step(self, event: StepEvent, result: object) -> None:
        metrics.record("step.completed", tags={
            "step": event.step_name,
            "workflow": event.workflow_name,
        })

    async def on_step_error(self, event: StepEvent, error: Exception) -> None:
        metrics.record("step.failed", tags={
            "step": event.step_name,
            "attempt": event.attempt,
        })

    async def on_workflow_error(self, event: WorkflowEvent, error: Exception) -> None:
        alerting.notify(f"Workflow {event.workflow_name} failed: {error}")

# Register BEFORE initialize()
app.register_plugin(MetricsPlugin())
await app.initialize()

Retry Configuration

Per-step RetryConfig with exponential or linear backoff. Step-level config overrides workflow defaults.

from pgflows import RetryConfig

# Per-step retry
@app.step(retry=RetryConfig(
    max_retries=5,                  # additional attempts after first failure
    backoff="exponential",          # "exponential" or "linear"
    initial_delay_seconds=2.0,
    max_delay_seconds=60.0,
    jitter=True,                    # randomize delay by +/-50%
))
async def my_step(ctx, input: MyInput) -> MyOutput: ...

# Workflow-level defaults (applied to all steps unless overridden)
@app.workflow(step_defaults=RetryConfig(max_retries=2))
async def my_workflow(ctx, input: MyInput) -> MyOutput: ...
FieldDefaultDescription
max_retries3Additional attempts after the first failure
backoff"exponential""exponential" or "linear"
initial_delay_seconds1.0Delay before the second attempt
max_delay_seconds60.0Cap on delay regardless of backoff factor
jitterTrueRandomize delay by +/-50% to avoid thundering herd

SQL Export

Export any registered Python workflow to a pg_durable SQL DSL string. Deploy definitions without code changes, or compose workflows at runtime from config.

from pgflows import SqlExporter

exporter = SqlExporter(
    registry=app.registry,
    base_url="https://api.example.com/pgflows"
)

# Export one workflow to runnable pg_durable SQL
sql = exporter.export_workflow("process_order")
# → SELECT df.setvar / df.start(df.http('.../steps/...') ~> ...)

# Dry-run: inspect steps without producing runnable SQL
result = exporter.dry_run("process_order")  # DryRunResult
for s in result.steps:
    print(s.step_name, s.http_url)

# Compose at runtime from a step list (no Python workflow fn needed)
sql = exporter.compose(
    workflow_name="on_call_response",
    steps=["page_engineer", "create_ticket", "apply_remediation"],
)

# Export all workflows (dev → prod migration)
sql_all = exporter.export_all()

# Select binding mode per export
http_sql   = app.exporter(base_url="https://...", mode="http").export_workflow("process_order")
worker_sql = app.exporter(mode="worker").export_workflow("process_order")
ℹ
AST-based step discovery
Step discovery relies on finding direct function references: await ctx.step(fn_name, inp) is detected. await ctx.step(lookup_fn(name), inp) is not.
ℹ
Two binding modes
mode="http" (default): pg_durable calls your FastAPI endpoint via df.http(). mode="worker": pg_durable enqueues via pgmq and a StepWorker runs the step.
✓
compose() validates step names
compose() checks that every step name is registered. Typos raise KeyError immediately rather than failing silently at runtime.

Cron Scheduling

PgCronBackend schedules recurring workflows via df.wait_for_schedule. Only the df extension is required; no pg_cron.

from pgflows import PgCronBackend

scheduler = PgCronBackend(dsn=config.dsn)
await scheduler.initialize()

# Schedule a workflow to run every hour
job_id = await scheduler.schedule(
    job_name="hourly_health_check",
    cron="0 * * * *",
    command="SELECT pgflows.enqueue_workflow('health_check', '{}')",
)

# List and manage jobs
jobs = await scheduler.list_jobs()         # list[ScheduledJob]
await scheduler.unschedule(job_id)

# Or use wait_for_schedule directly in a DSL graph:
from pgflows import wait_for_schedule, sql_node
node = (
    wait_for_schedule("0 * * * *")         # fires on the next cron tick
    >> sql_node("SELECT pgflows.enqueue_workflow('health_check', '{}')")
)
iid = await app.pg_durable.start(node, label="hourly-trigger")
ℹ
wait_for_schedule fires within ~1 min
The df cron evaluator checks at its own interval. A '* * * * *' schedule fires within about one minute in practice.
⚠
Chain wait_for_schedule from non-parallel nodes
Placing wait_for_schedule inside a parallel join or loop in the same instance can cause a ContinueAsNew deadlock on the bundled pg_durable build. Keep it in its own instance or at the start of a sequence.

Common Mistakes

Things that fail silently or with confusing errors the first time.

✗
Forgetting await app.initialize()
Must be called before app.start(), app.get_status(), or any other app method. Raises RuntimeError otherwise.
✗
app.pg_durable without checking availability
Accessing app.pg_durable raises RuntimeError if the df extension is absent. Check app.pg_durable_available first.
✗
Forgetting await app.close()
Connection pools stay open until the process exits. Always call app.close() in teardown or use it as an async context manager.
✗
Passing a string to app.start() instead of a function
app.start(process_order, inp) takes the function object. app.start("process_order", inp) raises AttributeError.
✗
DSL operator precedence
>> has lower precedence than & and |. Always wrap parallel branches explicitly: (a & b) >> c, not a & b >> c.
✗
setvar inside a push-mode step
Durable variables must be set before df.start() in SQL. From Python, call await client.setvar() before await client.start().
